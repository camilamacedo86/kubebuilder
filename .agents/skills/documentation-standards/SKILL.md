# Documentation Standards for Kubebuilder

Use this skill to write, review, or update Kubebuilder documentation following Kubernetes style guide standards.

**Documentation location**: `docs/book/src/` (mdBook format)

**Goal**: Enable high-quality, consistent documentation contributions that follow Kubernetes community standards and keep examples synchronized with tested code.

## Workflow

### For Writing New Documentation

1. Identify the documentation type (tutorial, plugin, reference, conceptual)
2. Use the appropriate template from [references/REFERENCE.md](references/REFERENCE.md)
3. **Verify technical accuracy**:
   - If documenting controller-runtime or controller-tools features, verify by checking source code
   - Use `go mod vendor` in temp directory to inspect actual implementation
   - See "Verifying Dependency Information" in [references/REFERENCE.md](references/REFERENCE.md)
4. Apply language standards (plain English, active voice, present tense, no contractions)
5. Apply formatting standards (sentence case, language tags, relative links)
6. **CRITICAL**: Include code from testdata using `{{#include}}` or `{{#literatego}}` shortcodes when available
   - Check if code exists in testdata first
   - Never copy/paste code inline if it exists in testdata
   - Use includes for maintainability and accuracy
7. Place link aliases at bottom of page
8. Review against checklist below

### For Reviewing Documentation

1. Check if editing auto-generated testdata (if yes, redirect to generators in `hack/docs/internal/`)
2. Scan for language issues:
   - Contractions -> expand
   - Passive voice -> active voice
   - Future tense -> present tense
   - "We" or "one" -> "you"
   - Lowercase kubernetes/yaml/json -> capitalize
3. Check formatting:
   - Code blocks without language tags
   - Absolute paths in links
   - Title case headings
4. Provide: issue, suggestion, reason (no "TIP!" callouts - use aside blocks)

### For Systematic Improvements

When improving docs at scale (like recent commits):
- Focus on one type of change per PR (contractions, capitalization, voice)
- Use consistent patterns across all files
- Test examples to ensure they still work
- Run `make fix-docs` before committing

## Core Principles

- Use plain English accessible to non-native speakers
- Follow Kubernetes documentation style guide
- Keep examples synchronized with tested code via testdata includes
- Prefer clarity over brevity
- No emojis or special characters

## Auto-Generated Testdata

Tutorial testdata projects under `docs/book/src/*/testdata/project/` are auto-generated.

**Critical**: Files in testdata are generated from two sources:

1. **Tutorial-specific generators** in `hack/docs/internal/`:
   - `internal/cronjob-tutorial/` - CronJob tutorial
   - `internal/getting-started/` - Getting Started guide
   - `internal/multiversion-tutorial/` - Multiversion tutorial

2. **Default scaffold from plugins** in `pkg/plugins/`:
   - `AGENTS.md` generated by `pkg/plugins/golang/v4/scaffolds/internal/templates/agents.go`
   - Other boilerplate files from their respective plugin templates

**To fix documentation issues in testdata**:
- If in tutorial-specific files: Edit generators in `hack/docs/internal/`
- If in default scaffold files (like `AGENTS.md`): Edit plugin templates in `pkg/plugins/`
- After template changes: Run `make install` to rebuild binary, then `make generate-docs`

See [references/REFERENCE.md](references/REFERENCE.md) for generator locations and commands.

## Auto-Generated Marker Documentation

**Marker documentation is auto-generated from controller-tools** and **cannot be edited directly**.

Files using `{{#markerdocs}}` preprocessor (auto-generated from controller-tools comments):
- `docs/book/src/reference/markers/crd.md`
- `docs/book/src/reference/markers/webhook.md`
- `docs/book/src/reference/markers/object.md`
- `docs/book/src/reference/markers/crd-validation.md`
- `docs/book/src/reference/markers/rbac.md`
- `docs/book/src/reference/markers/crd-processing.md`
- `docs/book/src/reference/controller-gen.md` (sections with `{{#markerdocs}}`)

**To fix marker documentation**:
1. Marker documentation comes from controller-tools repository, not Kubebuilder
2. Edit marker comments in controller-tools source code at `github.com/kubernetes-sigs/controller-tools`
3. Documentation is extracted from Go comments using the `markerdocs` preprocessor
4. The preprocessor is configured in `docs/book/book.toml` and runs via `docs/book/markerdocs.sh`

**Do NOT**:
- Edit marker documentation files directly in Kubebuilder
- Try to fix formatting or language issues in these generated sections
- Commit changes to auto-generated marker content

**If you find issues in marker documentation**: Report them to controller-tools repository.

## Language Standards

- **Plain English**: Avoid jargon, corporate speak, unnecessary complexity
  - Avoid: "leverage", "utilize", "facilitate", "in order to"
  - Use: "use", "help", "to"
- **Active voice**: Not passive
  - Good: "Kubebuilder generates the controller"
  - Bad: "The controller is generated by Kubebuilder"
- **Present tense**: Not future
  - Good: "The webhook validates the request"
  - Bad: "The webhook will validate the request"
- **Direct address**: Use "you", not "we" or "one"
  - Good: "Run the command to install"
  - Bad: "We can run the command" or "One should run the command"
- **No contractions**: Write out fully
  - Good: "do not", "it is", "you will"
  - Bad: "don't", "it's", "you'll"
- **Show, don't tell**: Use examples and demonstrations instead of abstract explanations
  - Good: Show code example with inline comments explaining behavior
  - Bad: Long paragraph describing what code should do

## Formatting Standards

- **Headings**: Use sentence case (`## Creating a new controller`)
- **Code blocks**: Must have language tags (` ```go ` not ` ``` `)
- **Inline code**: For commands, file names, API objects, field names
- **Internal links**: Use relative paths (`../guide.md` not `/docs/guide.md`)
- **Link aliases**: Always at bottom of page
- **Link text**: Descriptive (not "here" or "click here")
- **Admonitions**: Use `<aside class="note">` or `<aside class="warning">` (see [references/REFERENCE.md](references/REFERENCE.md))

## Code Examples

**ALWAYS prefer including from testdata when available** - do not copy/paste code inline.

### Why Use Testdata Includes

Using `{{#include}}` and `{{#literatego}}` shortcodes prevents documentation from becoming outdated:
- **Tested**: Examples are validated by `make test-book`
- **Maintained**: When code changes, docs auto-update
- **Accurate**: No drift between docs and reality
- **Trusted**: Users can rely on examples that actually work

### When to Use Includes vs Inline Code

**Use includes (PREFERRED)**:
- Tutorial code from `docs/book/src/*/testdata/project/`
- Plugin examples from `testdata/project-v4-with-plugins/`
- Any code that exists in a testdata project

**Use inline code (ONLY when)**:
- Very short snippets (1-3 lines)
- Pseudo-code showing a pattern
- Code that doesn't exist in testdata yet

### Include Shortcodes

```markdown
# For Go code (adds syntax highlighting, imports)
{{#literatego ./testdata/project/internal/controller/cronjob_controller.go}}

# For other files (YAML, JSON, shell)
{{#include ./testdata/project/config/manager/manager.yaml}}

# For specific sections with anchors
{{#include ./testdata/project/Makefile:docker-build}}
```

### Always Provide

1. Context before the code
2. Code include or inline block
3. Explanation after

Shell commands: Use `$` prefix for commands, no prefix for output.

**Critical**: All examples and commands MUST be tested and reproducible:
- Test shell commands in a clean environment (temp directory)
- Verify code examples compile and run
- Ensure commands produce the documented output
- **Prefer testdata includes over inline code** (already tested via `make test-book`)

See [references/REFERENCE.md](references/REFERENCE.md) for shortcode syntax and examples.

## Terminology

**Always capitalize in prose:**
- Kubernetes
- YAML, JSON, XML
- API, CRD, RBAC, HTTP, HTTPS
- Go (the language)

**Always lowercase in prose:**
- namespace, cluster, controller, webhook, pod, service
- kubectl, kubebuilder, kustomize
- docker, helm

**In code**: Exact capitalization as defined (`corev1.Namespace{}`, `metav1.ObjectMeta`)

## Common Mistakes to Avoid

When reviewing, watch for:
- **Documenting dependencies without verifying source code** -> use `go mod vendor` to check actual implementation
- **Inline code that should be included from testdata** -> use `{{#include}}` or `{{#literatego}}`
- Contractions (don't, can't, it's) -> expand them
- Future tense (will validate) -> present tense (validates)
- Passive voice (is generated by) -> active voice (generates)
- "We" or "one" -> "you" (direct address)
- Lowercase "kubernetes", "yaml", "json" -> capitalize
- Code blocks without language tags -> add ` ```go ` or ` ```bash `
- Absolute paths in links -> use relative paths
- "Click here" links -> descriptive link text
- Manual edits to auto-generated testdata -> edit generators instead
- Untested examples or commands -> test in clean environment first

**Most critical**: Using inline code when testdata includes exist = documentation will become outdated.

## Checklist

Before submitting:

- [ ] **Technical accuracy**: Verify controller-runtime/controller-tools features by checking source code
- [ ] **Code examples**: Use `{{#include}}` or `{{#literatego}}` from testdata (not inline copy/paste)
- [ ] Language: plain English, active voice, present tense, direct address, no contractions
- [ ] Terminology: Kubernetes, YAML, JSON capitalized; kubectl, namespace lowercase
- [ ] Formatting: sentence case headings, language tags on code blocks, aside blocks for notes
- [ ] Links: relative paths, aliases at bottom, descriptive text
- [ ] Code context: explanation before and after examples
- [ ] No manual edits to auto-generated testdata projects
- [ ] No emojis or special characters
- [ ] All commands tested in clean environment
- [ ] Run `make fix-docs` before committing
